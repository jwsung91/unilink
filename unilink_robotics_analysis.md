# Unilink 로봇 환경 (LiDAR/Depth 센서) 도입 분석 리포트

## 1. Unilink 벤치마크 결과 해석 (안정성 vs 데이터 완전성)

"Unilink가 예외(Exception)율이 0%이므로 안정성이 우세하다"는 주장은 절반만 맞습니다.
정확하게는 **"애플리케이션 레이어의 연결 관리 안정성"**이 우세한 것입니다.

*   **장점 (Unilink 우세):** 네트워크 단절 시 내부에서 투명하게 재연결을 처리하므로, 개발자는 복잡한 `try/except` 로직 없이 `send()`만 호출하면 됩니다. 일반적인 부하(LV1/LV2)에서는 거의 완벽한 전송률을 보장하며 트래픽 스루풋이 예측 가능합니다.
*   **단점 (Raw Socket 우세):** 극한 부하(LV3/LV4)에서는 비동기 I/O 오버헤드로 인해 최대 스루풋이 저하됩니다. 또한, 연결 단절 시 내부 큐에 쌓여있던 대량의 데이터(평균 7MB)가 한 번에 유실되는 단점이 있습니다.

---

## 2. 로봇 환경 (라이다, 뎁스 카메라) 적용 여부

원시(Raw) 형태의 라이다나 뎁스 카메라 데이터를 '실시간 제어용'으로 전송한다면 **현재 Unilink의 스펙은 부적합하며 튜닝이 필수적**입니다.

*   **대역폭 문제:** 센서 데이터는 초당 수 MB ~ 수십 MB가 쏟아집니다. 이는 Unilink가 처리하기 힘든 LV3/LV4 영역의 지속적인 부하를 유발합니다.
*   **지연(Latency) 문제 (치명적):** 로봇 제어에서는 "과거의 완벽한 데이터보다 현재의 약간 손실된 데이터가 낫습니다". Unilink는 데이터를 버리지 않고 큐에 쌓아두는 성향이 강해, 로봇의 인지 시스템이 수백 밀리초 이전의 과거 데이터를 보게 만드는 **지연 현상**을 초래합니다.
*   **권장 사용처:** Raw 센서 데이터 전송보다는, 엣지(Edge) 단에서 1차 가공된 가벼운 메타데이터(예: Bounding Box, 위치 좌표)나 끊기면 안 되는 **제어 명령(Command)** 전송에 매우 적합합니다.

## 3. 7MB 큐(Queue)의 원인과 튜닝 방안

벤치마크에서 확인된 약 7MB의 대규모 데이터 유실은 Unilink의 **백프레셔(Backpressure, 배압) 메커니즘** 때문입니다.
코드(`base/constants.hpp`) 상 기본 큐 허용치가 16MB(`DEFAULT_BACKPRESSURE_THRESHOLD`)로 설정되어 있어, 장애 발생 순간 큐에 머물던 데이터가 통째로 증발한 것입니다.

### 큐 크기를 대폭 줄일 경우 (예: 1MB 이하)
*   **장점:** 
    *   **지연 시간(Latency) 최소화:** 로봇이 가장 최신의 센서 데이터를 볼 수 있게 됩니다.
    *   연결 끊김 시 버려지는 데이터 양이 센서 프레임 1~2개 분량으로 최소화됩니다.
    *   메모리(RAM) 사용량이 제한되어 엣지 디바이스 환경에서 OOM 방지에 유리합니다.
*   **부작용:**
    *   네트워크가 조금만 지연되어도 큐 한계에 도달해 메시지가 바로 드랍(Drop)됩니다.
    *   큐라는 완충 지대가 사라져 전체 전송 스루풋이 요동치게 됩니다.
    *   백프레셔 이벤트(전송 중단/재개 알림)가 너무 빈번하게 발생하여 앱 레벨의 CPU 사용량이 늘어날 수 있습니다.

### 💡 로봇 센서용 튜닝 권장 사항
1. 초기화 시 `backpressure_threshold`를 **센서 프레임 1~3개 크기 수준(매우 작게)**으로 설정하세요.
2. 백프레셔(전송 중지) 알림이 오면 데이터를 큐에 밀어넣지 말고, 애플리케이션 단에서 최신 센서 프레임 1개만 덮어쓰기(Overwrite)하며 대기하는 로직을 구현해야 합니다.
